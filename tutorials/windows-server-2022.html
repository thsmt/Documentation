Windows Server 2022 手動構築手順

AWS EC2上にWindows Server 2022を手動構築する際の、インスタンス起動、接続、ローカルユーザー作成、OS初期設定、Windows Updateまでの確認観点を整理します。

前提 実環境名、顧客名、社内共有パス、実IPアドレスは扱わず、再利用できる手順として一般化しています。<INSTANCE_NAME>、<HOSTNAME>、<WINDOWS_PRIVATE_IP> などは環境に合わせて置き換えます。

1. 基本情報

Windows ServerはOSバージョンごとにメインストリームサポートと延長サポートの期限が異なるため、構築時点で利用するAMIとサポート期限を確認します。

OS メインストリーム終了 延長サポート終了 参考
Windows Server 2019 2024/01/09 2029/01/09 Microsoft Lifecycle
Windows Server 2022 2026/10/13 2031/10/14 Microsoft Lifecycle
Windows Server 2025 2029/10/09 2034/10/10 Microsoft Lifecycle

2. 作業前に決める値

インスタンス起動前に、以下の値をヒアリングシートや設計書で確認します。

値 用途
<REGION>構築リージョン。例: ap-northeast-1
<INSTANCE_NAME>EC2 Nameタグ。インスタンス、ボリューム、ENIへ同じ文脈で付与します。
<AMI_NAME> / <AMI_ID>Windows Server 2022 Japanese Full Baseなど、構築時点の対象AMI。
<INSTANCE_TYPE>要件に応じたEC2インスタンスタイプ。
<KEY_PAIR_NAME>Windows初期パスワード復号に使用するキーペア。
<VPC_ID> / <SUBNET_ID>配置先VPCとサブネット。
<SECURITY_GROUP_ID>RDP、踏み台、管理経路の要件に合わせたSecurity Group。
<IAM_INSTANCE_PROFILE>SSM、CloudWatch Agentなどに必要なIAMインスタンスプロファイル。
<HOSTNAME>OS上に設定するコンピューター名。
<ADMIN_USER> / <CUSTOMER_USER>管理用・利用者用のローカルユーザー名。

3. EC2インスタンス作成

3.1 名前とタグ

  1. AWS Management Consoleへ管理者権限でサインインします。
  2. 画面右上で対象リージョンを確認します。
  3. EC2ダッシュボードを開き、「インスタンスを起動」を選択します。
  4. 「さらにタグを追加」を選択します。
  5. Name キーに <INSTANCE_NAME> を設定し、リソースタイプはインスタンス、ボリューム、ネットワークインターフェイスを対象にします。

3.2 アプリケーションおよびOSイメージ

  1. 「その他のAMIを閲覧する」を選択します。
  2. 「コミュニティAMI」または利用するAMI提供元のタブを選択します。
  3. 検索欄に Windows_Server-2022-Japanese-Full-Base を入力します。
  4. 構築日時点で最新のWindows Server 2022日本語AMIを選択します。
  5. 検証環境と本番環境で同一AMIを使う場合は、AMI IDを作業記録に控えます。
確認 AWS提供のWindows AMIは定期的に更新されます。構築後に差分が追えるよう、AMI名とAMI IDを作業記録に残します。

3.3 インスタンスタイプ

設計値に合わせて <INSTANCE_TYPE> を選択します。T系インスタンスを利用する場合は、後続の高度な詳細でクレジット仕様も確認します。

3.4 キーペア

Windowsインスタンスの初期Administratorパスワード復号に使用するキーペアを選択します。Windows Serverの接続方式ではRSAキーペアを利用する運用になる場合があるため、環境の標準とAWSドキュメントを確認します。

  1. 既存の <KEY_PAIR_NAME> を選択、または新規作成します。
  2. 新規作成時は秘密鍵を安全な保管場所に保存します。
  3. 秘密鍵の保存場所、利用者、削除・ローテーション方針を作業記録に残します。

3.5 ネットワーク設定

  1. 設計値に合わせて <VPC_ID> と <SUBNET_ID> を選択します。
  2. Security Groupは <SECURITY_GROUP_ID> を選択します。
  3. Private Subnet配置の場合は、パブリックIPの自動割り当てを無効にします。

3.6 ストレージ

  1. 画面右側の「アドバンスト」を開き、ボリュームの詳細を展開します。
  2. ルートボリュームのサイズ、ボリュームタイプ、暗号化有無を確認します。
  3. 検証環境や一時サーバーでは、終了時に削除を「はい」に設定します。
  4. 追加ボリュームを作成する場合も、サイズ、タイプ、暗号化有無、終了時削除を確認します。
EBSの終了時削除 EC2起動後にEBSの DeleteOnTermination 属性を変更する場合は、AWS CLIの EC2 EBS終了時削除属性の変更 を利用します。

3.7 高度な詳細

  1. IAMインスタンスプロフィールに <IAM_INSTANCE_PROFILE> を選択します。
  2. 要件に応じて終了保護を有効化します。
  3. T2 / T3 / T3aを利用する場合は、クレジット仕様を確認します。
  4. 設定値を確認し、インスタンスを起動します。

4. Windowsインスタンス接続

接続方法は、踏み台経由のRDP、またはAWS Systems Manager Fleet Manager Remote Desktopを利用します。環境に応じて接続経路、IAM権限、ルート、Security Groupを確認します。

4.1 踏み台経由での接続

  1. 踏み台サーバーへSSHログインします。
  2. Tera TermなどのSSHクライアントで、SSHポートフォワード設定を開きます。
  3. ローカルポートに、他と重複しない任意のポートを設定します。例: 65001
  4. リモート側ホストに <WINDOWS_PRIVATE_IP> を指定します。
  5. リモート側ポートに 3389 を指定します。
  6. RDPクライアントで 127.0.0.1:65001 に接続します。
  7. ユーザー名 Administrator と、キーペアから復号したパスワードでログインします。
項目設定例
ローカルポート65001
リッスン未入力
リモート側ホスト<WINDOWS_PRIVATE_IP>
リモート側ポート3389
RDP接続先127.0.0.1:65001

4.2 Fleet Manager Remote Desktopでの接続

  1. EC2ダッシュボードで対象インスタンスを選択します。
  2. 「接続」から「RDPクライアント」を開きます。
  3. 接続タイプで「Fleet Managerを使用して接続する」を選択します。
  4. 認証種類にユーザー認証情報を選択します。
  5. ユーザー名 Administrator と、キーペアから復号したパスワードを入力します。
接続できない場合 パスワード未復号、SSM Agent未起動、IAMインスタンスプロファイルの権限不足、NAT GatewayまたはSSM関連VPCエンドポイントへの経路不足を確認します。IAMには AmazonSSMManagedInstanceCore 相当の権限が必要です。

4.3 Windowsパスワード取得

  1. 起動したWindowsインスタンスを選択します。
  2. 「アクション」から「セキュリティ」へ進み、「Windowsパスワードを取得」を選択します。
  3. プライベートキーファイルをアップロードします。
  4. 「パスワードを復号化」を選択します。
  5. 表示されたAdministratorパスワードを、認証情報管理ルールに従って安全に保管します。

5. ローカルユーザー作成

5.1 管理用ローカルユーザー

パスワードをコマンド履歴に残さないため、管理用ユーザーはGUIで作成します。

  1. Windows + R を押し、compmgmt.msc を実行します。
  2. 「ローカル ユーザーとグループ」から「ユーザー」を開きます。
  3. 「新しいユーザー」を選択します。
  4. ユーザー名、フルネーム、説明に <ADMIN_USER> を設定します。
  5. 「ユーザーは次回ログオン時にパスワードの変更が必要」のチェックを外します。
  6. 「パスワードを無期限にする」を選択します。
  7. 作成後、対象ユーザーの所属グループに Administrators を追加します。
  8. Administrators と Users に所属していることを確認します。

5.2 利用者用ローカルユーザー

利用者用ユーザーをPowerShellで作成する場合は、パスワードを平文で直接書かず、入力プロンプトからSecureStringとして受け取ります。

管理者権限でPowerShellを起動します。

powershell
net user

パスワードを対話入力で受け取り、ローカルユーザーを作成します。

powershell
$Password = Read-Host "Password for <CUSTOMER_USER>" -AsSecureString
New-LocalUser -Name "<CUSTOMER_USER>" -Password $Password -Description "<CUSTOMER_USER>" -PasswordNeverExpires

必要なローカルグループへ追加します。

powershell
Add-LocalGroupMember -Group "Administrators" -Member "<CUSTOMER_USER>"
Add-LocalGroupMember -Group "Users" -Member "<CUSTOMER_USER>"

作成結果を確認します。

powershell
net user <CUSTOMER_USER>

PowerShell履歴を削除します。

powershell
Clear-History
Remove-Item (Get-PSReadLineOption).HistorySavePath -ErrorAction SilentlyContinue

6. Windows OS初期設定

6.1 ホスト名設定

  1. Windows + R を押し、sysdm.cpl を実行します。
  2. 「コンピューター名」タブの「変更」を選択します。
  3. コンピューター名に <HOSTNAME> を入力します。
  4. ドメイン参加が必要な場合は、要件に応じてドメイン設定を行います。
  5. 再起動確認が表示された場合は、他の設定完了後に再起動します。

6.2 タイムゾーン設定

  1. Windows + R を押し、timedate.cpl を実行します。
  2. 「タイムゾーンの変更」を選択します。
  3. (UTC+09:00) 大阪、札幌、東京 を選択します。
  4. 現在時刻が日本時間になっていることを確認します。

6.3 Windows Update自動更新の無効化

運用上のメンテナンスウィンドウに合わせるため、自動更新を無効化し、更新は計画作業として実施します。

  1. Windows + R を押し、gpedit.msc を実行します。
  2. 「コンピューターの構成」から「管理用テンプレート」へ進みます。
  3. 「Windowsコンポーネント」から「Windows Update」を開きます。
  4. 「自動更新を構成する」を開きます。
  5. 「無効」を選択し、設定が無効になっていることを確認します。

6.4 RDP接続数の制限設定

  1. Windows + R を押し、gpedit.msc を実行します。
  2. 「管理用テンプレート」から「Windowsコンポーネント」を開きます。
  3. 「リモート デスクトップ サービス」から「リモート デスクトップ セッション ホスト」へ進みます。
  4. 「接続」を開き、「接続数を制限する」を無効にします。
  5. 「リモート デスクトップ サービス ユーザーに対してリモート デスクトップ サービス セッションを1つに制限する」も要件に応じて無効化します。

6.5 Windows Firewall設定

OS標準のWindows Firewallを無効化する場合は、Security Group、Network ACL、EDR、社内ポリシーで必要な制御が担保されていることを確認します。

  1. Windows + R を押し、firewall.cpl を実行します。
  2. 「Windows Defender ファイアウォールの有効化または無効化」を開きます。
  3. プライベート、パブリックの両方で無効化します。
  4. 「詳細設定」からプロパティを開きます。
  5. ドメイン、プライベート、パブリックの各プロファイルでファイアウォール状態を確認します。
注意 Windows Firewallの無効化は標準化された運用要件がある場合のみ実施します。公開サーバーや不特定経路から到達可能なサーバーでは、原則としてOS側ファイアウォールも含めて防御層を確認します。

6.6 追加ボリュームの初期化

ルートボリューム以外の追加ディスクをアタッチしている場合のみ実施します。

  1. Windows + R を押し、diskmgmt.msc を実行します。
  2. 未初期化ディスクが表示された場合、対象ディスクを確認します。
  3. パーティションスタイルは GPT を選択します。
  4. 未割り当て領域を右クリックし、「新しいシンプルボリューム」を選択します。
  5. ボリュームサイズは最大値、ドライブ文字は要件に応じて割り当てます。例: D:
  6. ファイルシステムは NTFS、アロケーションユニットサイズは既定値、クイックフォーマットを選択します。
  7. ディスクの状態が「正常」となり、エクスプローラーに対象ドライブが表示されることを確認します。

6.7 Windows Updateの実施

  1. Windows + R を押し、ms-settings:windowsupdate を実行します。
  2. 「更新プログラムのチェック」を選択します。
  3. 更新プログラムが検出された場合は、ダウンロードとインストールを実施します。
  4. 再起動が必要な場合は、作業影響を確認して再起動します。
  5. 再ログイン後、再度更新チェックを行い、「最新の状態です」と表示されるまで繰り返します。

7. CloudWatch Agent

EC2インスタンス作成後、要件に応じてCloudWatch Agentを導入し、メトリクスとログの収集設定を行います。

確認項目内容
IAM権限CloudWatch Agentがメトリクスとログを送信できる権限をIAMインスタンスプロファイルに付与します。
ログ収集Windows Event Log、アプリケーションログ、任意ログファイルの収集対象を確認します。
KMSロググループ暗号化が必要な場合は、CloudWatch LogsとKMSキーの設定を確認します。
再起動後確認サービス起動状態とCloudWatch側の取り込み状況を確認します。

8. 作業後確認

観点確認内容
EC2インスタンス名、AMI、インスタンスタイプ、サブネット、Security Group、IAMロールが設計値と一致していること。
接続踏み台経由またはFleet Manager経由でRDP接続できること。
ユーザー管理用・利用者用ローカルユーザーが作成され、必要なグループに所属していること。
OSホスト名、タイムゾーン、Windows Update、RDP設定、Firewall設定、追加ボリュームが要件通りであること。
監視CloudWatch Agentなどの監視・ログ収集設定が必要な場合、正常に送信されていること。
証跡設定値、スクリーンショット、実施日時、実施者、確認者を作業記録に残すこと。
