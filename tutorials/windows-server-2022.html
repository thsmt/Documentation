[構築] Windows Server 2022

ℹ️ 前提 実環境名、IPアドレス、顧客固有値は扱わず、再利用できる手順として一般化しています。<WINDOWS_PRIVATE_IP>、<HOSTNAME>、<ADMIN_USER>などは環境に合わせて置き換えます。

0. 起動時の確認

0.1 AMIとキーペア

AWSが提供するWindows Server 2022 AMIを選択し、秘密鍵を安全に管理できるRSAキーペアを指定します。

⚠️ 注意 Amazon EC2のWindowsインスタンスはED25519キーペアに対応していません。Administratorの初期パスワードを復号できるよう、Windows ServerではRSAキーペアを使用します。

AWS公式: CreateKeyPair

0.2 構築値

値用途
<WINDOWS_PRIVATE_IP>RDP接続先のプライベートIPアドレス。
<KEY_PAIR_NAME>Administrator初期パスワードの復号に使用するRSAキーペア。
<HOSTNAME>Windows Serverへ設定するコンピューター名。
<ADMIN_USER>管理用ローカルユーザー名。
<CUSTOMER_USER>利用者用ローカルユーザー名。
<VOLUME_DRIVE_LETTER>追加ボリュームへ割り当てるドライブ文字。例: D:

1. Windowsインスタンス接続

1.1 Administratorパスワード取得

  1. Amazon EC2コンソールで対象インスタンスを選択。
  2. 「アクション」から「セキュリティ」、「Windowsパスワードを取得」を選択。
  3. 起動時に指定したRSAキーペアの秘密鍵をアップロード。
  4. 「パスワードを復号化」を選択。
  5. 表示されたAdministratorパスワードを認証情報管理ルールに従って保管。

1.2 踏み台経由のRDP接続

  1. 踏み台サーバーへSSH接続。
  2. SSHクライアントでローカルポートフォワードを設定。
  3. ローカルポートへ未使用の任意ポートを指定。例: 65001
  4. 転送先へ<WINDOWS_PRIVATE_IP>:3389を指定。
  5. リモートデスクトップクライアントで127.0.0.1:65001へ接続。
  6. Administratorと復号したパスワードでサインイン。

1.3 Fleet Manager Remote Desktop接続

  1. AWS Systems ManagerのFleet Managerで対象のマネージドノードを選択。
  2. 「ノードアクション」から「リモートデスクトップで接続」を選択。
  3. ユーザー認証情報を使用する接続方式を選択。
  4. Administratorと復号したパスワードで接続。
ℹ️ 情報 Fleet Managerを利用するEC2インスタンスには、SSM AgentとAmazonSSMManagedInstanceCore相当の権限を持つIAMインスタンスプロファイルが必要です。インスタンスへアクセスキーを保存せず、IAMロールで権限を付与します。

AWS公式: Systems Managerに必要なインスタンス権限

2. ローカルユーザー作成

2.1 管理用ローカルユーザー

  1. Windows + Rを押し、compmgmt.mscを実行。
  2. 「ローカル ユーザーとグループ」から「ユーザー」を開く。
  3. 「新しいユーザー」を選択。
  4. ユーザー名へ<ADMIN_USER>を入力。
  5. パスワード運用要件に合わせて初回変更と有効期限を設定。
  6. 対象ユーザーをAdministratorsグループへ追加。

2.2 利用者用ローカルユーザー

管理者権限でPowerShellを起動し、パスワードを対話入力で受け取ります。

powershell
$Password = Read-Host "Password for <CUSTOMER_USER>" -AsSecureString
New-LocalUser -Name "<CUSTOMER_USER>" -Password $Password -Description "<CUSTOMER_USER>"

業務上必要な最小権限のローカルグループへ追加します。

powershell
Add-LocalGroupMember -Group "Users" -Member "<CUSTOMER_USER>"
⚠️ 注意 Administratorsグループへの追加とパスワード無期限化は、対象ユーザーに管理権限が必要で、社内標準で許可されている場合に限ります。Active Directoryへ参加する環境では、ローカルアカウントを増やさず、ドメインアカウントとグループポリシーによる管理を優先します。

3. Windows OS初期設定

3.1 ファイル名拡張子の表示

  1. エクスプローラーを開く。
  2. 「表示」から「表示」を開く。
  3. 「ファイル名拡張子」を有効化。

実行ファイルや設定ファイルの種類を誤認しないよう、拡張子を常に表示します。

3.2 コンピューター名設定

  1. Windows + Rを押し、sysdm.cplを実行。
  2. 「コンピューター名」タブから「変更」を選択。
  3. コンピューター名へ<HOSTNAME>を入力。
  4. ドメイン参加が必要な場合は、名前解決と接続先Directory Serviceを確認して設定。
  5. 再起動要求を保留し、他の再起動を伴う設定とまとめて実施。

3.3 タイムゾーン設定

  1. Windows + Rを押し、timedate.cplを実行。
  2. 「タイムゾーンの変更」を選択。
  3. (UTC+09:00) 大阪、札幌、東京を選択。
ℹ️ 情報 2018年8月以降のAWS提供Windows AMIは、Amazon Time Sync Serviceを既定で使用します。Windows Server 2022の標準AMIでは追加のNTP設定は不要です。ドメイン参加後は、Active Directoryの時刻階層を含む設計に従います。

AWS公式: Amazon Time Sync Service

3.4 Windows Update自動更新の無効化

計画したメンテナンスウィンドウで更新を実施する運用に合わせ、自動更新を無効化します。

  1. Windows + Rを押し、gpedit.mscを実行。
  2. 「コンピューターの構成」、「管理用テンプレート」、「Windowsコンポーネント」、「Windows Update」の順に開く。
  3. 「自動更新を構成する」を開く。
  4. 「無効」を選択して保存。
⚠️ 注意 自動更新を無効化する場合は、AWS Systems Manager Patch Manager、WSUS、SCCMなどで定期的に更新を適用する運用を必ず用意します。更新経路がない状態で無効化すると、既知の脆弱性が残ります。

3.5 RDPセキュリティ設定

3.5.1 ネットワークレベル認証

  1. Windows + Rを押し、sysdm.cplを実行。
  2. 「リモート」タブを開く。
  3. 「ネットワーク レベル認証でリモート デスクトップを実行しているコンピューターからのみ接続を許可する」を有効化。

ネットワークレベル認証は、RDPセッションを確立する前に利用者を認証し、未認証接続によるリソース消費と攻撃対象領域を抑えます。

Microsoft公式: Remote Desktopとネットワークレベル認証

3.5.2 RDPセッション制限

  1. Windows + Rを押し、gpedit.mscを実行。
  2. 「管理用テンプレート」、「Windowsコンポーネント」、「リモート デスクトップ サービス」、「リモート デスクトップ セッション ホスト」、「接続」の順に開く。
  3. 「接続数を制限する」を開き、運用要件に合わせて設定。
  4. 複数セッションが必要な場合のみ、「リモート デスクトップ サービス ユーザーに対してリモート デスクトップ サービス セッションを1つに制限する」を無効化。

3.6 Windows Defender Firewallの無効化

本手順の対象環境では、通信制御をSecurity Groupなどの上位レイヤーへ集約し、OS側との二重管理による設定差分や疎通障害を避ける社内標準に基づいてWindows Defender Firewallを無効化します。

  1. Windows + Rを押し、wf.mscを実行。
  2. 「Windows Defender ファイアウォールのプロパティ」を開く。
  3. ドメイン、プライベート、パブリックの各プロファイルで「ファイアウォールの状態」を「無効」に変更。
  4. 設定を保存。
⚠️ 注意 Windows Defender Firewallの無効化は、すべての環境に適用する一般的なAWS推奨ではありません。AWSはOS側ファイアウォールを多層防御として利用することを推奨しています。本手順は、Security GroupとNetwork ACLによる通信制御、EDRによる検知・防御、変更承認が整備された対象環境に限って適用します。

AWS公式: Windowsインスタンスのセキュリティベストプラクティス

3.7 追加ボリュームの初期化

  1. Windows + Rを押し、diskmgmt.mscを実行。
  2. 未初期化ディスクの容量とディスク番号を確認。
  3. パーティションスタイルへGPTを選択。
  4. 未割り当て領域から「新しいシンプル ボリューム」を選択。
  5. ドライブ文字へ<VOLUME_DRIVE_LETTER>を割り当て。
  6. ファイルシステムへNTFSを選択し、要件に合わせたボリュームラベルを設定。
  7. クイックフォーマットを実行。

3.8 Windows Update

  1. Windows + Rを押し、ms-settings:windowsupdateを実行。
  2. 「更新プログラムのチェック」を選択。
  3. 検出された更新プログラムをインストール。
  4. 再起動要求がある場合は、作業影響を確認して再起動。
  5. 更新プログラムが検出されなくなるまで繰り返し実行。

4. AWS管理機能の設定

4.1 CloudWatch Agent導入

AWS Systems Manager Run Commandを使用して、Amazon CloudWatch Agentをインストールします。

  1. AWS Systems Managerコンソールで「Run Command」を開く。
  2. コマンドドキュメントへAWS-ConfigureAWSPackageを選択。
  3. ActionへInstall、NameへAmazonCloudWatchAgentを指定。
  4. 対象インスタンスをタグまたはインスタンスIDで指定。
  5. コマンドを実行。
  6. 収集するWindowsイベントログとパフォーマンスカウンターを定義した設定ファイルを、Systems Manager Parameter Storeまたは承認済みの配布経路から適用。
ℹ️ 情報 CloudWatch Agentを使用する場合は、EC2インスタンスのIAMロールへCloudWatchAgentServerPolicy相当の権限を付与します。設定ファイルに認証情報を保存しません。

AWS公式: CloudWatch Agentのインストール

4.2 EC2Launch v2とAMI横展開

AWS提供のWindows Server 2022 AMIにはEC2Launch v2が含まれます。構築済みインスタンスをカスタムAMIとして横展開する場合は、EC2Launch v2からSysprepを実行してインスタンス固有情報を一般化します。

  1. 構築作業とWindows Updateを完了。
  2. 一時ファイル、個別の認証情報、作業用アカウント、固定IP設定などを除去。
  3. スタートメニューから「EC2Launch settings」を開く。
  4. AMI起動後に必要なタスクとSSM Agentの起動設定を確認。
  5. 「Shutdown with Sysprep」を実行。
  6. インスタンス停止後にAmazon EC2コンソールからAMIを作成。
⚠️ 注意 Sysprepはバックアップ手段ではありません。コンピューター名やSIDなどのインスタンス固有情報を一般化するため、稼働中インスタンスの復旧用AMIを取得する目的では使用しません。

AWS公式: EC2Launch v2設定

5. 要件に応じた追加設定

5.1 TLS 1.0 / 1.1の無効化

組織の暗号化基準でTLS 1.2以上を必須とする場合は、IIS、.NET Framework、データベースクライアント、監視エージェントなどの互換性を検証してからTLS 1.0 / 1.1を無効化します。

⚠️ 注意 OS全体のSchannel設定を変更すると、古いミドルウェアや外部システムとの通信が失敗することがあります。対象プロトコルを使用するアプリケーション、切り戻し用レジストリ値、再起動影響を確認してから適用します。

Microsoft公式: WindowsにおけるTLS 1.0 / 1.1の非推奨化

5.2 イベントログとクラッシュダンプ

監視要件と障害解析要件に合わせて、Windowsイベントログの最大サイズと上書き方式、ページファイル、クラッシュダンプの保存方式を設計します。ローカルログだけに依存せず、CloudWatch Logsなどの集中管理先へ転送します。

ℹ️ 情報 完全メモリダンプには、OSボリューム上へ十分なページファイル容量が必要です。メモリ容量、OSボリュームの空き容量、障害時に必要な解析レベルを確認し、システム管理サイズまたは組織標準の値を使用します。

Microsoft公式: クラッシュダンプ設定の推奨事項
